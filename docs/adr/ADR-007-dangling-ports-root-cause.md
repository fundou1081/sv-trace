# ADR-007: 悬空端口检测 & 根因分析器

> **状态**: In Progress
> **日期**: 2026-04-19
> **版本**: v0.1

---

## 背景

DebugAssistant 需要更深入的问题检测能力:
1. **悬空端口检测**: 发现未连接的模块端口
2. **根因分析器**: 给定症状，推断根本原因

---

## Part 1: 悬空端口检测

### 问题定义

找出模块中没有被任何信号连接的端口 (input/output/inout)

### 算法方案

#### Option A: 直接匹配 (Direct Matching)

步骤:
1. 获取模块所有端口列表
2. 从实例化获取连接列表
3. 匹配端口名与连接目标
4. 输出未匹配的端口

**优点**:
- 实现简单直接
- 速度快 O(n)

**缺点**:
- 只能检测显式实例化的连接
- 不考虑隐式连接 (如 .* 端口映射)

**复杂度**: O(n)

---

#### Option B: 类型感知检测 (Type-Aware Detection)

步骤:
1. 端口分类: input → 需要驱动, output → 需要负载
2. 检查 input 是否被驱动
3. 检查 output 是否有负载
4. 递归检查子模块

**优点**:
- 考虑端口方向
- 更准确判断"悬空"

**缺点**:
- 需要分析信号驱动情况
- 稍复杂

**复杂度**: O(n × m) where m = signals

---

#### Option C: 多层级分析 (Hierarchical Analysis)

步骤:
1. 收集所有模块实例
2. 对每个实例:
   a. 获取端口列表
   b. 获取连接映射
   c. 递归检查子模块
3. 考虑层级路径

**优点**:
- 支持跨层级检测
- 准确处理generate/数组实例

**缺点**:
- 最复杂
- 性能可能较差

**复杂度**: O(n²)

---

### 推荐方案

**选择 Option B (类型感知检测)**

理由:
- 比 A 更准确 (考虑方向)
- 比 C 简单且足够用
- 平衡准确性和性能

---

## Part 2: 根因分析器

### 问题定义

给定一个症状 (X值/问题信号)，找出最可能的根本原因

### 症状类型

1. **X值传播** - 信号值为 X 或 Z
2. **多驱动冲突** - 同一信号被多次驱动
3. **未初始化** - 寄存器未复位
4. **时序违规** - 路径延迟过大

### 算法方案

#### Option A: 反向追溯 (Backward Tracing)

步骤:
1. 从问题信号开始
2. 递归查找驱动源
3. 检查每个驱动是否可能产生问题
4. 评分排序可能原因

**优点**:
- 方向正确 (从症状到原因)
- 直观易懂

**缺点**:
- 可能路径过长
- 难以量化概率

**复杂度**: O(d^k) where d=avg drivers, k=depth

---

#### Option B: 正向传播 (Forward Propagation)

步骤:
1. 收集所有潜在问题源 (未初始化/多驱动/条件未覆盖)
2. 从每个问题源正向传播
3. 检查是否能到达目标信号
4. 标记可达的问题源

**优点**:
- 能发现隐藏问题
- 可以提前预警

**缺点**:
- 可能产生误报
- 需要完整的问题源列表

**复杂度**: O(p × n) where p=problem sources

---

#### Option C: 双向分析 (Bidirectional)

步骤:
1. 反向: 从症状追溯可能的驱动路径
2. 正向: 从已知问题源检查可达性
3. 交叉分析: 找交集中的最可能原因
4. 置信度评分

**优点**:
- 最全面
- 准确度最高

**缺点**:
- 最复杂
- 性能开销大

**复杂度**: O(n²)

---

#### Option D: 规则引擎 (Rule-Based)

步骤:
1. 定义问题模式规则:
   - 规则1: 多驱动 → 条件分支未覆盖
   - 规则2: 未初始化 → 缺少复位分支
   - 规则3: X值传播 → 条件未覆盖的case
2. 匹配代码模式
3. 返回匹配规则

**优点**:
- 快速简单
- 易于理解和调试
- 可扩展规则库

**缺点**:
- 依赖规则完整性
- 可能漏掉复杂情况

**复杂度**: O(r × n) where r=rules

---

### 推荐方案

**选择 Option D (规则引擎) 作为 Phase 1**

理由:
- 实现快速
- 足够处理常见问题
- 易于扩展

**后续可增强 Option C (双向分析)**

## 算法对比总结

### 悬空端口检测

| 算法 | 复杂度 | 准确性 | 实现难度 |
|------|--------|--------|----------|
| A: 直接匹配 | O(n) | 低 | 简单 |
| B: 类型感知 | O(n×m) | 中 | 中等 |
| C: 多层级 | O(n²) | 高 | 复杂 |

### 根因分析器

| 算法 | 复杂度 | 准确性 | 可扩展性 |
|------|--------|--------|----------|
| A: 反向追溯 | O(d^k) | 中 | 难 |
| B: 正向传播 | O(p×n) | 中 | 中 |
| C: 双向 | O(n²) | 高 | 难 |
| D: 规则引擎 | O(r×n) | 中 | 易 |

---

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 规则不完整 | 提供规则扩展接口 |
| 误报多 | 置信度评分，低置信度不报告 |
| 性能差 | 增量分析 + 缓存 |
