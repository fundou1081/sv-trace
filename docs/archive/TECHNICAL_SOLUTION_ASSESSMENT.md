## 一、技术方案分类框架

### 1.1 分类标准

```
┌─────────────────────────────────────────────────────────────────────┐
│                    技术方案评估维度                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  编译工具型 (Compiler/Build)                                         │
│  ├── 需要深度集成 (仿真器、综合器、版本控制)                         │
│  ├── 大规模数据处理 (代码解析、波形分析)                             │
│  ├── 确定性输出 (同样输入必须产生同样结果)                           │
│  └── 示例: Lint检查、代码生成、覆盖率统计                            │
│                                                                     │
│  编译+Skill混合型 (Hybrid)                                           │
│  ├── 工具执行 + 领域知识判断                                        │
│  ├── 规则引擎 + LLM推理                                             │
│  └── 示例: 变更影响分析、CDC协议检查                                │
│                                                                     │
│  纯Skill/领域知识型 (Knowledge)                                      │
│  ├── 文本处理、模板、清单                                            │
│  ├── 规则判断、知识库查询                                            │
│  ├── LLM辅助生成和推理                                               │
│  └── 示例: Spec模板、检查清单、文档生成                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 分类原则

| 类型 | 特征 | 适用场景 |
|------|------|----------|
| **编译工具** | 确定性、可重复、计算密集 | 数据管理、静态分析、报告解析 |
| **Skill** | 知识型、模板化、规则化 | 文档生成、清单检查、经验判断 |
| **Hybrid** | 工具+知识、两者结合 | 分析+判断、解析+理解 |

        {features}

### 8.2 Rule Skill模式

```yaml
skill:
  name: feature-prioritizer
  type: rule
  description: Feature优先级判断
  
  prompts:
    - when: "用户需要确定Feature优先级"
      rule_engine: |
        Feature分级规则:
        
        P0 (Must):
        - 影响芯片是否能工作的功能
        - 客户明确要求的功能
        - 竞品有的核心功能
        
        P1 (Should):
        - 显著提升产品竞争力的功能
        - 团队强烈建议的功能
        - 降低风险的功能
        
        P2 (Could):
        - 优化用户体验的功能
        - 未来可能需要的功能
        - 资源允许时做的功能
        
        P3 (Won't):
        - 复杂度高价值低的功能
        - 未来版本再考虑的功能
        
        请根据以上规则分析:
        {features}
```

### 8.3 Hybrid Skill模式

```yaml
skill:
  name: signal-classifier
  type: hybrid
  description: 信号分类与可视化
  
  tools:
    - svtrace.rtl_parser       # 解析RTL
    - svtrace.signal_extractor # 提取信号
  
  prompts:
    - when: "用户需要理解RTL信号"
      workflow: |
        Step 1: 解析RTL
        ```python
        parser = SVParser()
        parser.parse_file("{file}")
        signals = extract_signals(parser)
        ```
        
        Step 2: 分类信号
        根据领域知识，对信号进行分类:
        
        Clock信号:

        - 数据通路上的信号

        Control信号:
        - 控制逻辑信号

        Port信号:
        - 模块端口
        - 按输入/输出分类

### 9.2 关键洞察

**洞察1**: **超过一半的需求 (54%) 是Skill型**
```

```
关系: Skill调用编译工具

例子: 信号分类

```

```
