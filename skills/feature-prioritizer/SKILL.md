# Feature Prioritizer Skill

功能优先级判断

## 功能

根据预定义规则，自动对功能需求进行优先级分级。

## 分级规则

### P0 - Must Have (必须)
满足以下任一条件：
- 影响芯片基本功能
- 客户明确要求
- 竞品已有功能
- 安全性相关
- 不实现芯片无法工作

### P1 - Should Have (应该)
满足以下条件：
- 显著提升产品竞争力
- 团队强烈建议
- 降低风险
- 显著提升性能
- 重要但可workaround

### P2 - Could Have (可以)
满足以下条件：
- 优化用户体验
- 未来可能需要
- 资源允许时实现
- 复杂度低价值高

### P3 - Won't Have (暂不)
满足以下条件：
- 复杂度高价值低
- 未来版本考虑
- 与路线图冲突
- 客户未要求

## 判断流程

```
1. 分析Feature描述
2. 对照P0条件 → 命中? → P0
3. 对照P1条件 → 命中? → P1
4. 对照P2条件 → 命中? → P2
5. 默认 → P3
```

## 输入格式

```markdown
Feature: 支持自动波特率检测
描述: UART接收器自动检测9600-115200波特率
原因: 客户反馈需要适配不同设备
影响: 需要新增3个寄存器，修改RX路径
```

## 输出格式

```markdown
# Feature优先级分析

## Feature: 支持自动波特率检测

## 评估结果: P1 (Should Have)

## 分析:
| 条件 | 匹配 | 说明 |
|------|-------|------|
| 影响基本功能 | ❌ | 手动配置可替代 |
| 客户明确要求 | ✅ | 客户反馈需要 |
| 竞品已有 | ❌ | 未知 |
| 显著竞争力 | ✅ | 差异化功能 |
| 复杂度 | 中 | 需新增3个寄存器 |

## 理由:
客户明确要求，但手动配置波特率可作为workaround，
因此归类为P1。

## 建议:
1. MVP可暂时不实现
2. 规划到下一版本
3. 准备手动配置方案
```

## 示例分析

### Example 1
```
Feature: 支持115200波特率
评估: P0
理由: 基本功能，不支持则UART无意义
```

### Example 2
```
Feature: FIFO深度从16扩展到32
评估: P1
理由: 提升性能，但不是Must
```

### Example 3
```
Feature: 支持IrDA红外协议
评估: P2
理由: 特殊场景需要，通用性不强
```

### Example 4
```
Feature: 支持USB接口
评估: P3
理由: 超出当前模块范围，应作为独立IP
```

## 决策树

```
新Feature
    │
    ├─── 是否影响芯片正常工作?
    │       YES → P0
    │       NO
    │           ├─── 是否客户明确要求?
    │           │       YES → P0
    │           │       NO
    │           │           ├─── 是否显著提升竞争力?
    │           │           │       YES → P1
    │           │           │       NO
    │           │           │           ├─── 实现复杂度低?
    │           │           │           │       YES → P2
    │           │           │           │       NO → P3
```
