# Spec转验证点 AI策略框架

## 1. 目的

将自然语言描述的规格文档(Spec)自动转换为可执行的验证测试点(Verification Points)。

## 2. 方法论

### 2.1 输入

```
Spec文档 (Markdown/Word/PDF)
├── 功能描述
├── 接口定义
├── 时序要求
├── 边界条件
└── 异常场景
```

### 2.2 输出

```
Verification Points (结构化测试点)
├── 功能点 (Functional)
│   ├── 正常路径
│   ├── 边界条件
│   └── 异常处理
├── 时序点 (Timing)
│   ├── 建立时间
│   └── 保持时间
└── 协议点 (Protocol)
    ├── 握手时序
    └── 错误检测
```

## 3. 转换流程

```
┌─────────────┐
│  Spec输入   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  文本解析   │  ← NLP/正则
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  实体识别   │  ← LLM/NER
│  - 功能    │
│  - 参数    │
│  - 条件    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  关系抽取   │  ← LLM/RE
│  - 因果    │
│  - 依赖    │
│  - 约束    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  测试点生成 │  ← LLM/Generation
│  - 场景   │
│  - 约束   │
│  - 预期   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  验证点输出 │
└─────────────┘
```

## 4. Prompt模板

### 4.1 实体识别Prompt

```markdown
你是一个芯片验证工程师。
请从以下Spec中识别所有功能实体:

Spec:
{text}

识别:
1. 功能模块
2. 输入参数及其范围
3. 输出及其含义
4. 控制信号
5. 状态变量

以JSON格式输出:
{
  "modules": [...],
  "parameters": [...],
  "signals": [...]
}
```

### 4.2 测试点生成Prompt

```markdown
你是一个验证工程师。
根据以下Spec片段生成验证测试点:

Spec片段:
{text}

要求:
1. 生成正常路径测试
2. 生成边界条件测试
3. 生成异常场景测试
4. 每个测试包含:
   - 测试名称
   - 前置条件
   - 输入
   - 预期输出
   - 优先级

输出格式:
| 测试名称 | 前置条件 | 输入 | 预期输出 | 优先级 |
```

## 5. 实现策略

### 5.1 方案A: 纯LLM

```
优点:
- 实现简单
- 泛化能力强
- 无需训练

缺点:
- 成本高
- 延迟大
- 一致性差

适用场景:
- 演示/原型
- 小规模Spec
```

### 5.2 方案B: LLM+RAG

```
优点:
- 利用已有知识
- 一致性好
- 可增量学习

缺点:
- 需要知识库
- 实现复杂

适用场景:
- 量产使用
- 大规模Spec
```

### 5.3 方案C: Fine-tuned Model

```
优点:
- 精度高
- 速度快
- 一致性好

缺点:
- 需要标注数据
- 训练成本高

适用场景:
- 稳定项目
- 长期使用
```

## 6. 质量评估

### 6.1 评估指标

| 指标 | 计算方法 | 目标 |
|------|----------|------|
| 召回率 | 生成的测试点/人工标注 | >90% |
| 准确率 | 正确的测试点/生成总数 | >85% |
| 完整性 | 覆盖Spec条款数/总条款数 | >95% |

### 6.2 人工审核

- 生成的测试点需人工审核
- 关键测试必须验证
- 建立反馈机制

## 7. 使用建议

1. **初期**: 使用方案A (纯LLM) 快速验证
2. **中期**: 积累数据，转方案B (RAG)
3. **后期**: Fine-tune专属模型
