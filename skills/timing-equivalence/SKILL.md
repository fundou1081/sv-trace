# 时序等效性检查 AI策略框架

## 1. 目的

验证RTL优化(如流水线、retiming)前后的功能等效性。

## 2. 方法论

### 2.1 挑战

```
优化可能引入:
- 流水线级数变化
- 延迟改变
- 握手协议变化
```

### 2.2 检查策略

```
┌─────────────┐
│ 原始RTL    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 功能提取   │  ← AI/Logic
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 行为模型   │  ← LLM/FSM
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 优化RTL    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 功能提取   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 等效性验证 │  ← 形式验证
└───────────┘
```

## 3. 检查方法

### 3.1 功能等效

```
方法:
1. 提取两个RTL的FSM
2. 比较状态转移
3. 比较输出响应

工具:
- LLM分析FSM
- 形式验证 (Conformal)
```

### 3.2 时序等效

```
方法:
1. 识别流水线级数
2. 补偿延迟
3. 验证输出对齐

工具:
- STA报告分析
- 波形对比
```

## 4. 实现策略

### 4.1 方案A: 形式验证

```
优点:
- 数学证明
- 无需仿真

缺点:
- 需专业工具
- 复杂度高

工具:
- Conformal EC
- VC Formal
- JasperGold
```

### 4.2 方案B: 仿真+LLM

```
优点:
- 实现简单
- 灵活

缺点:
- 不完整

方法:
1. 生成定向测试
2. 对比输出
3. LLM分析差异
```

### 4.3 方案C: 混合

```
标准流程:
1. 形式验证基本等效
2. 仿真验证corner case
3. LLM分析边界差异
```

## 5. AI辅助流程

### 5.1 LLM辅助分析

```markdown
# FSM比较Prompt

你是一个验证工程师。
请比较两个FSM是否等效:

FSM1:
{fsm1}

FSM2:
{fsm2}

检查:
1. 状态数是否相同
2. 转移条件是否等价
3. 输出是否一致
4. 延迟差异是否可接受

输出:
{
  "equivalent": true/false,
  "differences": [...],
  "risk_level": "high/medium/low"
}
```

## 6. 检查清单

- [ ] FSM状态一致
- [ ] 转移条件等价
- [ ] 输出延迟可补偿
- [ ] 无功能差异
- [ ] Corner case覆盖
